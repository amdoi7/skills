# 设计哲学与 DDD 对照

## 核心目标

软件设计的核心问题不是"把功能写出来"，而是持续管理复杂度。DDD 不是替代这件事，而是把复杂度管理提升到业务语言、边界和协作层面。

## 复杂度三症状

1. **Change Amplification**：小改动牵动很多位置。
2. **Cognitive Load**：为了改一个点，必须先理解大量上下文。
3. **Unknown Unknowns**：根本不知道还会影响哪里。

## 把《软件设计哲学》翻译到 DDD 语境

| 设计哲学概念 | DDD 落点 | 实操含义 |
| --- | --- | --- |
| Information Hiding | 限界上下文 | 上下文内部模型自己演化，对外只暴露契约 |
| Deep Module | 聚合 / 聚合根 | 外部看到少量清晰行为，复杂规则留在边界内部 |
| Pull Complexity Downward | 充血模型 / 值对象 | 不把一致性和校验扩散到所有调用方 |
| Define Errors Out of Existence | 值对象 / 语义化接口 | 用类型和语义消灭一部分非法状态 |
| Good Naming / Consistency | 统一语言 | 文档、代码、接口共享稳定术语 |
| Design It Twice | 聚合划分 / 上下文划分 | 重要边界至少比较两种方案 |

## BDD / TDD 作为设计反馈

- **BDD** 负责暴露"对外必须可观察到什么行为"。
- **TDD** 负责暴露"这个行为最自然应该由哪个公开接口承担"。
- 如果一个场景测试必须穿透很多内部层，或者为了隔离行为要 mock 一串内部对象，边界通常切错了。
- 如果同一条规则在多个测试层面重复断言，说明知识可能已经泄漏。

## 建模判断

- 先问"哪些规则必须一起保持一致"，再问"要几个类"。
- 先问"这个词在当前边界里到底是什么意思"，再问"数据库怎么存"。
- 先把不变式放回聚合和值对象，再决定是否需要领域服务。
- 应用服务主要回答"先做什么后做什么"，不应该成为业务规则仓库。
- Repository 负责用领域语言取和存聚合，不是机械包装 DAO。

## 设计红旗

- **Shallow Module**：类很多，但每层只转发一下。
- **Pass-Through Method**：方法几乎只是转调下一层。
- **Pass-Through Variable**：同一个参数被层层透传，中间层却不理解它。
- **Temporal Decomposition**：只是按执行顺序切模块，而不是按知识和边界切。
- **Information Leakage**：同一条规则出现在多个类、多个层、多个测试里。
- **Vague Name**：`data`、`info`、`manager`、`util` 之类名字太多。
- **Comment Repeats Code**：注释只是翻译代码，没有解释意图、边界和约束。

## 命名与注释

- 名字要表达业务语义，不要只表达技术角色。
- 接口注释写"做什么、为什么、边界条件"，少写实现细节。
- 如果一个概念很难命名，通常不是词汇问题，而是边界还没想清楚。

## 设计复盘问题

1. 这个设计让最常见的使用路径更简单了吗？
2. 复杂度被模块吸收了，还是被配置、调用方、流程代码承担了？
3. 同一条规则是否只在一个地方被真正守住？
4. 这个行为能否通过一个稳定公开接口被自然测试？
5. 读者只看接口和少量注释，能否明白模块职责？
