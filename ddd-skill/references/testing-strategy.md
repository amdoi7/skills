# BDD / TDD 测试策略

## 非妥协原则

1. **测试行为而非实现细节**：围绕对外接口与业务结果写测试，内部重构不应迫使测试改动。
2. **Mock 系统边界而非内部层级**：前端只 mock 网络请求，后端只 mock 第三方服务，不对 `router/service/repository` 逐层 mock。
3. **优先内存数据库而非数据库 API mock**：验证真实查询与约束行为，减少脆弱断言。
4. **不纠结测试分类标签**：在可接受的耗时与稳定性前提下，优先提升发布信心。
5. **覆盖率是手段不是目标**：重点是暴露隐藏 bug、死代码与冗余分支。

## TDD 不是只写测试，它也是切边界的方法

把"哪个行为该放在哪个边界里"交给 TDD 来验证：

### 倾向留在同一边界的信号

- 这些场景共享同一套业务词汇。
- 这些场景共享同一条不变式或事务一致性要求。
- 这些场景最自然地通过同一个公开接口来测试。
- 这些场景失败时返回的是同一类业务语义。

### 倾向拆成不同边界的信号

- 这些场景需要不同的外部依赖和不同的 fixture。
- 这些场景已经开始使用不同的统一语言。
- 为了测试其中一个行为，你不得不初始化大量无关状态。
- 为了隔离行为，你开始 mock 内部层级而不是系统边界。

## BDD 到 TDD 的落地顺序

1. 先从用户故事写出 `Given / When / Then` 场景。
2. 为每个场景选一个最自然的公开测试入口。
3. 写第一个失败测试。
4. 如果测试入口不自然，回头重切边界。
5. 让聚合、值对象或用例处理这个行为通过。
6. 重构内部设计，但保持场景测试稳定。

## 推荐测试组合

### 场景测试

- 目标：验证主路径、关键异常和业务结果。
- 入口：HTTP、命令处理器、应用服务或公开 use case。

### 领域行为测试

- 目标：验证聚合和值对象的不变式、状态流转、领域事件。
- 写法：直接通过公开行为调用对象，不测私有方法。

### 持久化测试

- 目标：验证 Repository 的真实查询、唯一约束、事务边界、映射正确性。
- 默认：优先 SQLite 内存库、Testcontainers 或临时真实数据库。

### 系统边界测试

- 目标：验证与支付、短信、邮件、对象存储等外部依赖的交互。
- 默认：只 mock 这些边界，断言契约与关键副作用。

## 反模式

- 为了隔离而把内部层每一层都 mock 掉。
- 针对私有方法、内部 helper 或调用次数写大量脆弱断言。
- 因为测试分类焦虑，把高信号测试拆碎成很多低信号小测试。
- 只追求覆盖率数字，忽略测试是否真正覆盖风险。
- 用"单测一定不能碰数据库"这种标签替代对反馈速度和真实度的判断。

## 设计回馈

测试不是设计之后的附属物。好的测试策略会反过来暴露：

- 哪些公开接口太薄，像 pass-through；
- 哪些规则没有真正收敛在领域边界内部；
- 哪些上下文切分让场景测试变得笨重；
- 哪些异常其实可以通过更好语义"定义到不存在"。
