# Paragraph Structure

段落层面的 AI 写作模式。核心问题：把段落当装饰而不是论证单元。

## 碎片段落

过度使用极短句或句子碎片作为独立段落来制造虚假的强调感。

RLHF 训练把模型推向"每句一个想法"的最低公分母可读性。没有真人这样写初稿，因为这不符合人思考和说话的方式。

坏例子：

```
He published this. Openly. In a book. As a priest.
```

```
These weren't just products.
And the software side matched.
Then it professionalised.
But I adapted.
```

```
Platforms do.
```

修正方式：把相关的短句合并成完整段落。一个段落应该发展一个想法，包含支撑细节，而不是把每个词都当成炸弹投下去。

合并后："他作为牧师，在一本公开出版的书里写了这些"——一句话就够了。

## 判断标准

什么时候短段落是合理的：
- 对话体写作中模拟说话节奏
- 在长段落之后做一次真正的转折
- 段落确实只有一个不可再分的论点

什么时候短段落是 AI 腔：
- 连续 3 个以上单句段落
- 每个段落都在"揭示"而不在论证
- 去掉段落间的换行后读起来更流畅

## 变相列表（Listicle in a Trench Coat）

本质是列表，但包装成连续散文。每个段落以"The first... The second... The third..."开头。

可能是因为你告诉模型别生成列表，它就把列表伪装成了段落。

坏例子：

```
The first wall is the absence of a free, scoped API...

The second wall is the lack of delegated access...

The third wall is the absence of scoped permissions...
```

修正方式：
- 如果内容确实是并列的，就用真正的列表——不丢人
- 如果内容有逻辑递进关系，就写成真正的散文，用因果/转折连接段落，而不是用序数词

## 段落的基本职责

每个段落应该满足至少一条：
1. 推进一个新论点或新证据
2. 对前一段做实质性的反驳或限定
3. 用具体案例支撑前一段的抽象论点

不应该出现的段落类型：
- 纯过渡段（"现在让我们看看另一个方面"）
- 纯重申段（用不同的词重复上一段说的话）
- 纯气氛段（只营造情绪不传递信息）

## 自查规则

1. 检查是否有连续 3 个以上单句段落——合并或扩展它们
2. 检查段落开头是否用了序数词（第一、第二、首先、其次）——如果是，考虑改成列表或重写连接逻辑
3. 对每个段落问："去掉这段后文章少了什么？"如果答案是"什么也没少"，删掉这段
